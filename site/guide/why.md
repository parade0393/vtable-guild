# 为什么选择 vtable-guild

vtable-guild 不是一个全新表格心智的库，而是给已经在使用 ant-design-vue 或 element-plus 的项目提供一条更顺手的替换路径。

它的目标很明确：保留你熟悉的列定义、排序筛选和业务集成方式，同时补上原生表格在大数据量、列布局控制和主题扩展上的短板。

## 目标用户

这套库更适合以下场景：

- 你的项目已经使用 ant-design-vue 或 element-plus，希望表格视觉与整体 UI 体系保持一致。
- 你不想为虚拟滚动、列宽拖拽、条纹行和 hover 行状态继续维护额外封装。
- 你需要一套能在全局、业务线和单实例三层同时控制样式的表格主题机制。
- 你希望迁移成本可控，而不是把现有表格页面全部推倒重来。

## 它解决的核心问题

### 1. 原表格能力不够用

在业务表格逐渐复杂之后，常见痛点往往不是“能不能渲染表格”，而是：

- 大数据量场景下滚动体验差。
- 列宽需要用户现场调整，但原表格与业务封装之间常常还要再补一层约定。
- 条纹行、hover 行、边框等视觉状态要靠额外 class 或 CSS 覆盖。
- 同一套业务代码需要接入不同视觉体系时，样式调整成本高。

### 2. 继续在原表格上叠补丁会越来越重

如果你已经围绕 ant-design-vue Table 写了很多业务代码，最昂贵的部分通常不是 JSX 或模板，而是列配置、数据处理和页面交互约定。

vtable-guild 的价值在于尽量保留这部分使用方式，把增强能力做在表格内部，而不是把复杂度继续压给业务层。

## 相比直接继续使用原表格

### 对 ant-design-vue 用户

vtable-guild 的主要优势在于增强能力，而不是完全改写 API 习惯。

- 常见列配置、排序和筛选写法仍然熟悉。
- 内置虚拟滚动，而 ant-design-vue Table 不提供这项能力。
- 滚动条体验针对 antdv 预设做了额外打磨，优化方向参考了 element-plus。
- 列宽控制继续沿用接近 antdv 的字段心智，striped、hoverable、ui 覆盖也都可以直接作为一等能力使用。
- 主题覆盖不是靠散落样式补丁，而是通过 preset、theme 和 ui 分层完成。

### 对 element-plus 用户

如果你更看重统一的表格行为与风格切换能力，vtable-guild 的价值在于把 element-plus 普通 Table 里偏向 el-table-column 的组织方式，收敛成 columns + props 的统一模型，再配合主题预设和覆盖机制一起使用，而不是为每个项目重新拼装能力。

## 什么时候不适合

如果你的项目只需要一个非常简单的数据展示表，没有性能、拖拽、主题切换或复杂交互需求，那么继续使用现有 UI 库自带表格通常更省事。

vtable-guild 更适合中大型业务表格，而不是为了“替换而替换”。
